Overview

Stats service provides the following functionality:

  • Updates of in-memory stats objects on requests from motr nodes
  • Provides stats information from in-memory stats objects to cluster administrative utilities/console on their query.

Definitions

  • Stats Service Stats service which processes motr statistics update/query requests.
  • Stats Object Stats in-memory object which represents a statistics matrix.
  • Stats Update Stats update request which updates stats object values.
  • Stats Query Stats query request which returns latest updated stats objects value.
  • Stats Clients Motr administrative console/utilities.

Requirements

  • r.stats_service.in_memory_objects Maintains in-memory statistic objects. It is list of stats object.
  • r.stats_service.update It updates requested stats object values.
  • r.stats_service.query It returns requested stats object values to stats clients.

Logical Specification

Stats Object List

In memory stats object list does not contain any object initially. Stats FOM update respective stats object. object is created if not found in stats list.

Threading and Concurrency Model

Stats service FOMs(updates/query) are executed by request handler's locality thread. Since same locality is assigned to all these FOMs, no other serialization is required for stats service.

Service Registration

Stats service type definition :

struct m0_reqh_service_type m0_stats_svc_type = { .rst_name = "M0_CST_STATS", .rst_ops = &stats_service_type_ops, .rst_level = M0_RS_LEVEL_NORMAL, .rst_typecode = M0_CST_STATS, };

Stats service type initialization/finalization :

Stats service registers/unregisters its service type with request handler using interfaces m0_stats_svc_init()/ m0_stats_svc_fini() during Motr system initialization and finalization (m0_init()/ m0_fini()).

NUMA optimizations

Update FOMs and query FOMs are executed in the same locality. Update/Query FOMs access in-memory list of statistic objects. Since both type of FOMs executed by same localiy threads it gets benifis of locality.

Dependencies

  • r.reqh : Request handler's locality thread execute stats service FOMs(update/query).

Conformance

  • i.stats_service.in_memory_objects It implements list of stats object to keep stats run time.
  • i.stats_service.update It implements stats_update FOM.
  • i.stats_service.query It implements stats_query FOM.

  4. Test query for stats object with single stats parameter

Integration Test

Motr nodes send stats (different 2-3 types) to stats service, verification of stats sent by Motr node with respective stats object.

System Test

This tests whole monitoring infrastructure by running some sample IOs on on Motr and verify statistics collected by querying statistic service using client/monitoring utility.

Analysis

  • Since all the stats objects are in-memory objects, query and update are fast.
  • Since stats service is just a book keeping of stats information into in-memory object list. Since it does not do any computations of stats values, it does not creates CPU usage overhead for motr nodes.
